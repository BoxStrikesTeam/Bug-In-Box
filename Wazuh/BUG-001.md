# Overview

During a manual source code review of the Wazuh File Integrity Monitoring (FIM) module, BoxStrike identified a missing NULL pointer validation in the `fim_whodata_event()` function.

The issue occurs because the `inode` and `dev` members of the `whodata_evt` structure are passed directly to `strtoul()` without verifying that they contain valid pointers.

Under specific conditions, this may result in a Null Pointer Dereference and cause the Wazuh agent to terminate unexpectedly.

The finding was responsibly reported to the Wazuh maintainers.

---

# Understanding the Bug

The vulnerable code performs the following conversions:

```c
const unsigned long inode = strtoul(w_evt->inode, NULL, 10);
const unsigned long device = strtoul(w_evt->dev, NULL, 10);
```

If either `w_evt->inode` or `w_evt->dev` is `NULL`, the program invokes:

```c
strtoul(NULL, NULL, 10);
```

which leads to undefined behavior and typically results in a segmentation fault.

---

# Root Cause

The root cause is straightforward:

* `w_evt->inode` is assumed to be valid.
* `w_evt->dev` is assumed to be valid.
* Neither pointer is validated before use.

This violates a fundamental defensive programming principle:

> Never dereference or pass pointers to library functions without first verifying that they are valid.

---

# Impact Assessment

From a software engineering perspective, this is a genuine implementation bug.

Under specific conditions, it can lead to:

* Process crash
* Segmentation Fault
* Unexpected agent termination
* Reduced service reliability

---

# Maintainer Assessment

After reviewing the report, the Wazuh maintainers confirmed that the missing NULL checks exist.

However, they concluded that the issue does **not** represent a practical security vulnerability because triggering the vulnerable state would require an attacker to already possess elevated (root or kernel-level) privileges, such as by injecting crafted audit events through a malicious eBPF program or another trusted event source.

For this reason:

* ❌ No CVE was assigned.
* ✅ The report was accepted as a robustness / defense-in-depth issue.
* ✅ A follow-up issue was created to add the missing NULL checks.
* ✅ The researcher received official credit for identifying the bug.

---

# Recommended Fix

The recommended remediation is to validate both pointers before calling `strtoul()`.

Example:

```c
const unsigned long inode =
    w_evt->inode ? strtoul(w_evt->inode, NULL, 10) : 0;

const unsigned long device =
    w_evt->dev ? strtoul(w_evt->dev, NULL, 10) : 0;
```

This simple validation prevents undefined behavior and improves the robustness of the FIM component.

# Lessons Learned

Not every programming error becomes a security vulnerability.

Sometimes, a coding flaw is acknowledged and corrected because it improves software reliability rather than because it enables a realistic attack.

This case highlights the importance of distinguishing between **code correctness** and **practical exploitability**. Even when a bug does not warrant a CVE, identifying and fixing such issues contributes to more resilient software.
